# 게시글 목록 오류 상태별 화면 동작

## 1. 목록 조회 오류

| HTTP 상태 | `message` | UI 동작 | 데이터 유지 | 이동 |
|---:|---|---|---|---|
| 400 | `invalid_request` | `게시글 목록 요청값을 다시 확인해주세요.` alert | 기존 카드 유지 | 없음 |
| 401 | `unauthorized_request` | `로그인이 필요합니다.` alert | 의미 없음 | `/login` |
| 500 | `internal_server_error` | `서버 오류가 발생했습니다. 잠시 후 다시 시도해주세요.` alert | 기존 카드 유지 | 없음 |
| 그 외 | 구분하지 않음 | `게시글 목록을 불러오는 중 오류가 발생했습니다.` alert | 기존 카드 유지 | 없음 |

오류가 끝나면 요청 잠금 상태를 해제한다. 따라서 sentinel이 다시 교차하거나 재시도 동작이 추가되면 같은 커서로 다시 요청할 수 있다.

## 2. 네트워크·응답 처리 오류

다음 오류는 status가 없는 일반 오류로 처리한다.

- 네트워크 연결 실패
- CORS 차단
- JSON으로 파싱할 수 없는 응답
- 예상 응답 필드가 없어 렌더링 중 발생한 오류

표시 문구:

```text
게시글 목록을 불러오는 중 오류가 발생했습니다.
```

## 3. 사용자 정보 조회 오류

| 조건 | Vanilla 동작 |
|---|---|
| 401 | alert 없이 로그인 페이지로 이동 |
| 그 외 | console에만 기록하고 현재 화면 유지 |

React에서는 `ProtectedRoute`와 AuthContext 인증 초기화가 미인증 진입을 우선 처리한다. 이미 렌더링된 뒤 사용자 정보 요청이 401인 경우에도 최종 결과는 `/login` 이동으로 유지한다.

## 4. 토큰 재발급 오류

공통 API client가 Access Token을 지운 뒤 오류를 다시 전달한다. 현재 Vanilla에서는 호출 위치에 따라 다음처럼 보일 수 있다.

- 목록 요청에서 전달되면 일반 목록 오류 alert
- 사용자 조회에서 전달되면 status에 따라 로그인 이동 또는 console 기록

React에서는 재발급 실패를 AuthContext의 미인증 상태로 동기화하고 `/login`으로 보내는 방향을 권장한다. 이는 화면별 오류 분기의 불일치를 제거하는 설계 보완이며 구현 시 변경 기록에 남긴다.

## 5. 로그아웃 오류

| 조건 | UI 동작 | 이동 |
|---|---|---|
| 모든 실패 | `로그아웃 중 오류가 발생했습니다. 다시 시도해주세요.` alert | 없음 |

- 요청 중에는 로그아웃 버튼을 비활성화한다.
- 실패 후 버튼을 다시 활성화한다.
- 실패 시 Access Token을 임의로 제거하지 않는다.

## 6. 빈 목록은 오류가 아님

- `posts`가 빈 배열이면 alert를 표시하지 않는다.
- 기존 Vanilla에는 빈 상태 안내가 없으므로 카드 영역이 비어 보인다.
- React에서 `등록된 게시글이 없습니다.` 같은 UI를 추가하려면 사용자 경험 개선으로 구분하고 Vanilla 동등성 검증 항목과 분리한다.

## 7. 추가 조회 실패 주의사항

- 이미 표시된 게시글을 초기화하지 않는다.
- 커서를 성공 전까지 변경하지 않는다.
- 실패한 페이지의 일부 데이터를 임의로 추가하지 않는다.
- 중복 요청 잠금을 반드시 해제한다.
- 별도의 하단 오류 UI나 재시도 버튼은 현재 Vanilla에 없다.
