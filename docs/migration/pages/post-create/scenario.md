# 게시글 작성 정상 작동 시나리오

## 1. 기준 소스

- 화면: `pages/make-post.html`
- 작성 동작: `js/make-post.js`
- Header 동작: `js/profile-menu.js`
- 검증: `js/validation.js`
- 공통 요청: `js/common.js`
- 스타일: `css/common.css`, `css/make-post.css`

## 2. 페이지 정보

- 기존 URL: `pages/make-post.html`
- React 예정 URL: `/posts/new`
- 진입 대상: 로그인 사용자
- 뒤로가기·로고 이동: `/posts`
- 작성 성공 이동: 생성된 ID가 있으면 `/posts/:postId`, 없으면 `/posts`

`/posts/new`는 `/posts/:postId`보다 명확한 정적 Route로 등록한다. 상세 화면에서 `new`를 잘못된 게시글 ID로 해석하지 않도록 Router 선언과 검증에서 구분한다.

## 3. 화면 구성

1. 목록 뒤로가기, 서비스 로고, 프로필 메뉴가 있는 Header
2. `게시글 작성` 페이지 제목
3. 최대 26자의 게시글 제목 input
4. 게시글 내용 textarea
5. 여러 이미지 파일 선택 input
6. 선택한 파일명과 이미지 미리보기 목록
7. 화면 하단에 고정된 `완료` 버튼

## 4. 초기 진입

1. 보호 Route가 현재 사용자 인증을 초기화한다.
2. 인증에 성공하면 공통 Header와 빈 작성 폼을 표시한다.
3. 제목·내용이 비어 있으므로 완료 버튼은 비활성화한다.
4. 이미지 영역에는 `선택된 이미지가 없습니다.`를 표시한다.

## 5. 제목과 내용 검증

### 제목

- 앞뒤 공백을 제거한 값이 비어 있으면 `* 제목을 입력해주세요.`를 표시한다.
- 앞뒤 공백을 제거한 길이가 26자를 초과하면 `* 제목은 최대 26자까지 작성 가능합니다.`를 표시한다.
- HTML `maxlength="26"`도 입력 길이를 제한한다.
- 유효하면 제목 helper를 비운다.

### 내용

- 앞뒤 공백을 제거한 값이 비어 있으면 `* 내용을 입력해주세요.`를 표시한다.
- 유효하면 내용 helper를 비운다.
- 현재 프론트엔드에는 내용 최대 길이 제한이 없다.

### 검증 시점

- 입력 중에는 제목과 내용의 유효성을 계산해 완료 버튼만 갱신한다.
- 각 필드에서 blur가 발생하면 해당 helper를 갱신한다.
- 제출 시 두 필드를 모두 다시 검증한다.
- 제목과 내용이 모두 유효하고 작성 요청 중이 아닐 때만 완료 버튼을 활성화한다.

## 6. 이미지 선택과 미리보기

1. file input은 `accept="image/*"`, `multiple` 조건을 사용한다.
2. 선택이 바뀌면 이전 Object URL을 모두 `URL.revokeObjectURL()`로 반환한다.
3. 새 `FileList`를 배열로 저장한다.
4. 파일이 없으면 `선택된 이미지가 없습니다.`를 표시하고 미리보기를 비운다.
5. 파일이 있으면 파일명을 쉼표로 연결해 표시한다.
6. 각 파일에 Object URL을 만들고 파일명 기반 alt를 가진 미리보기 이미지를 표시한다.
7. 페이지를 떠날 때 남은 Object URL을 모두 반환한다.

현재 Vanilla에는 파일 개수, 용량, 확장자와 미리보기 로드 실패에 대한 별도 UI 검증이 없다. 브라우저의 `accept`는 선택 편의를 위한 힌트이며 보안 검증으로 간주하지 않는다.

## 7. 게시글 작성 요청

1. submit 기본 동작을 막는다.
2. 이미 작성 요청 중이면 추가 제출을 무시한다.
3. 제목과 내용을 다시 검증하고 실패하면 요청하지 않는다.
4. 제목·내용의 앞뒤 공백을 제거한다.
5. 선택한 이미지의 파일명만 `imageUrls` 배열로 만든다.
6. 완료 버튼을 비활성화하고 `POST /posts`를 한 번 요청한다.
7. 성공 응답에 `data.postId`가 있으면 해당 상세 페이지로 이동한다.
8. `postId`가 없으면 목록으로 이동한다.
9. 실패하면 현재 입력값과 이미지 선택을 유지하고 오류 UI를 표시한다.
10. 이동하지 않은 경우 요청 상태를 해제하고 폼 유효성에 따라 버튼을 복구한다.

요청 예시는 다음과 같다.

```json
{
  "title": "오늘 배운 React",
  "content": "컴포넌트의 책임을 나누었다.",
  "imageUrls": ["preview-1.png", "preview-2.jpg"]
}
```

## 8. 공통 Header

- 뒤로가기와 로고는 게시글 목록으로 이동한다.
- 사용자 프로필 이미지를 표시하고 실패하면 기본 이미지를 사용한다.
- 프로필 버튼으로 회원 메뉴를 열고 닫는다.
- 회원정보 수정, 비밀번호 수정과 로그아웃 기능은 기존 `AppLayout`을 재사용한다.
- 로그아웃 성공 시 로그인 페이지로 이동한다.

## 9. React 마이그레이션에서 보존할 동작

- 보호 Route와 Access Token 재발급 정책
- 제목 26자, 제목·내용 trim 검증과 helper 문구
- blur와 submit 검증 시점
- 유효성과 요청 상태에 따른 완료 버튼 활성화
- 다중 이미지 선택, 파일명 표시와 미리보기
- 이미지 선택 변경·unmount 시 Object URL 반환
- JSON의 `imageUrls`에 파일명만 전송하는 현재 계약
- 요청 중 중복 제출 차단
- 성공 응답의 `postId` 유무에 따른 이동
- status/message별 helper, alert와 이동
- 공통 Header, 프로필 메뉴와 로그아웃
