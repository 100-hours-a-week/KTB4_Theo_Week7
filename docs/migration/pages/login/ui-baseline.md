# 로그인 텍스트 UI 기준선

이 문서는 이미지 캡처 대신 로그인 화면의 상태를 텍스트로 기록한다. React 마이그레이션 후 동일한 입력과 동작을 수행하고, 아래의 구조·문구·상태와 비교한다.

## 상태 표기 규칙

- `[ ]`: 입력 필드
- `[비활성]`: 선택하거나 제출할 수 없는 버튼
- `[활성]`: 선택하거나 제출할 수 있는 버튼
- `helper:`: 입력 필드 아래에 표시되는 안내 또는 오류 문구
- `alert:`: 브라우저 alert로 표시되는 문구
- `→`: 페이지 이동
- 비밀번호 입력값은 실제 값 대신 `••••••••`로 기록한다.

## 공통 화면 구조

```text
┌─────────────────────────────────────────────┐
│ </> 코드 한입                               │
│ 개발자의 성장 한입, 지식의 공유 한입       │
│                                             │
│ 로그인                                      │
│ 이메일로 로그인하고 오늘의 개발 이야기를   │
│ 이어가세요.                                 │
│                                             │
│ 이메일                                      │
│ [이메일을 입력하세요]                       │
│                                             │
│ 비밀번호                                    │
│ [비밀번호를 입력하세요]                     │
│                                             │
│ [비활성: 로그인]                            │
│                                             │
│ 아직 회원이 아니신가요? 회원가입            │
└─────────────────────────────────────────────┘
```

## UI-LOGIN-01: 최초 진입

### 조건

- 로그인 페이지에 처음 진입한다.
- 이메일과 비밀번호를 입력하지 않는다.

### 화면 상태

```text
이메일
[값: 빈 문자열]
helper: 표시 안 함

비밀번호
[값: 빈 문자열]
helper: 표시 안 함

[비활성: 로그인]
```

### 동작

- 이메일 필드의 자동완성 목적은 `email`이다.
- 비밀번호 필드의 자동완성 목적은 `current-password`다.
- `회원가입`을 선택하면 회원가입 페이지로 이동한다.

## UI-LOGIN-02: 이메일 빈 값 오류

### 조작

1. 이메일 필드에 포커스한다.
2. 값을 입력하지 않고 다른 위치로 포커스를 이동한다.

### 화면 상태

```text
이메일
[값: 빈 문자열]
helper: * 이메일을 입력해주세요.

비밀번호
[값: 빈 문자열]
helper: 표시 안 함

[비활성: 로그인]
```

## UI-LOGIN-03: 이메일 형식 오류

### 조작

1. 이메일에 `user`를 입력한다.
2. 이메일 필드에서 포커스를 이동한다.

### 화면 상태

```text
이메일
[값: user]
helper: * 올바른 이메일 주소 형식을 입력해주세요. (예: example@adapterz.kr)

비밀번호
[값: 빈 문자열]
helper: 표시 안 함

[비활성: 로그인]
```

## UI-LOGIN-04: 비밀번호 빈 값 오류

### 조작

1. 비밀번호 필드에 포커스한다.
2. 값을 입력하지 않고 다른 위치로 포커스를 이동한다.

### 화면 상태

```text
이메일
[값: 빈 문자열]
helper: 표시 안 함

비밀번호
[값: 빈 문자열]
helper: * 비밀번호를 입력해주세요.

[비활성: 로그인]
```

## UI-LOGIN-05: 비밀번호 형식 오류

### 조작

1. 비밀번호에 `password`를 입력한다.
2. 비밀번호 필드에서 포커스를 이동한다.

### 화면 상태

```text
비밀번호
[값: ••••••••]
helper: * 비밀번호는 8자 이상, 20자 이하이며, 대문자, 소문자, 숫자,
        특수문자를 각각 최소 1개 포함해야 합니다.

[비활성: 로그인]
```

## UI-LOGIN-06: 유효한 폼

### 조건

- 이메일 형식이 유효하다.
- 비밀번호가 모든 형식 조건을 만족한다.
- 로그인 요청 중이 아니다.

### 화면 상태

```text
이메일
[값: user@example.com]
helper: 표시 안 함

비밀번호
[값: ••••••••••]
helper: 표시 안 함

[활성: 로그인]
```

## UI-LOGIN-07: 로그인 요청 중

### 조작

1. 유효한 폼에서 로그인 버튼을 선택한다.
2. 서버 응답이 도착하기 전 상태를 확인한다.

### 화면 상태

```text
이메일
[입력값 유지]

비밀번호
[입력값 유지]

[비활성: 로그인]
```

### 동작

- `POST /auth/login` 요청이 진행 중이다.
- 버튼을 다시 선택하거나 폼을 다시 제출해도 추가 요청을 보내지 않는다.
- 별도의 로딩 문구나 spinner는 표시하지 않는다.

## UI-LOGIN-10: 요청값 오류

### 서버 응답

```text
status: 400
message: invalid_request 또는 정의되지 않은 message
```

### 사용자 피드백

```text
alert: 입력값을 다시 확인해주세요.
```

입력값은 유지되며 기존 helper는 모두 제거된다.

## UI-LOGIN-11: 로그인 시도 제한

### 서버 응답

```text
status: 429
message: too_many_login_attempts 또는 다른 message
```

### 사용자 피드백

```text
alert: 로그인 시도가 너무 많습니다. 잠시 후 다시 시도해주세요.
```

입력값은 유지되며 요청이 끝난 후 폼이 유효하면 버튼이 다시 활성화된다.

## UI-LOGIN-13: 네트워크 또는 기타 오류

### 조건

- 백엔드 연결 실패
- 파싱할 수 없는 응답
- 성공 응답에 Access Token이 없음


## UI-LOGIN-14: 로그인 성공

### 서버 응답



## React 비교 판정 기준

React 로그인 화면은 다음 조건을 만족하면 텍스트 기준선과 동등하다고 판정한다.

- 같은 조건에서 같은 필드 아래에 같은 helper 문구가 표시된다.
- 최초·유효·요청 중 상태의 버튼 활성 여부가 일치한다.
- 실패 후 입력값 유지 여부가 일치한다.
- API 상태와 message에 따른 alert 또는 helper 처리 방식이 일치한다.
- 요청 중 중복 제출이 발생하지 않는다.
- 로그인 성공 후 `/posts`로 이동한다.
- 화면 요소의 순서와 label 연결, 자동완성 목적이 유지된다.

색상, 픽셀 간격, 글꼴 크기처럼 텍스트로 정확히 비교하기 어려운 시각 요소는 기존 CSS를 1차 마이그레이션에서 그대로 사용해 보존한다.
