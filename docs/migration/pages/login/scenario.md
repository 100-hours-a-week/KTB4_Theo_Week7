# 로그인 페이지 정상 시나리오

## 로그인 페이지

### 기준 소스

- 화면: `pages/login.html`
- 동작: `js/login.js`
- 공통 요청: `js/common.js`
- 검증 규칙: `js/validation.js`

### 페이지 정보

- 기존 URL: `pages/login.html`
- 마이그레이션 예정 URL: `/login`
- 문서 제목: `로그인`
- 주요 목적: 이메일과 비밀번호로 인증하고 게시글 목록으로 이동한다.
- 진입 대상: 비로그인 사용자

### 화면 구성

1. 서비스 로고와 소개 문구
2. 이메일 입력 필드
3. 이메일 검증 메시지 영역
4. 비밀번호 입력 필드
5. 비밀번호 검증 메시지 영역
6. 로그인 버튼
7. 회원가입 페이지 링크

### 최초 진입 상태

- 이메일과 비밀번호 입력값은 비어 있다.
- 두 helper 영역에는 문구가 없다.
- 로그인 버튼은 `disabled` 상태다.
- 이메일 필드에는 `autocomplete="email"`이 설정되어 있다.
- 비밀번호 필드에는 `autocomplete="current-password"`가 설정되어 있다.

### 입력 및 검증 시나리오

#### 이메일

1. 사용자가 이메일을 입력한다.
2. 입력 중에는 전체 폼의 유효성을 계산해 로그인 버튼 상태를 갱신한다.
3. 이메일 입력 필드에서 포커스가 벗어나면 값을 `trim()`한 후 검증한다.
4. 빈 값이면 `* 이메일을 입력해주세요.`를 표시한다.
5. 이메일 형식이 아니면 `* 올바른 이메일 주소 형식을 입력해주세요. (예: example@adapterz.kr)`를 표시한다.
6. 유효하면 이메일 helper를 비운다.

이메일 형식은 `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` 정규식으로 판정한다.

#### 비밀번호

1. 사용자가 비밀번호를 입력한다.
2. 입력 중에는 전체 폼의 유효성을 계산해 로그인 버튼 상태를 갱신한다.
3. 비밀번호 입력 필드에서 포커스가 벗어나면 검증한다.
4. 빈 값이면 `* 비밀번호를 입력해주세요.`를 표시한다.
5. 형식이 맞지 않으면 비밀번호 규칙 안내 문구를 표시한다.
6. 유효하면 비밀번호 helper를 비운다.

비밀번호는 다음 조건을 모두 만족해야 한다.

- 8자 이상 20자 이하
- 영문 대문자 최소 1개
- 영문 소문자 최소 1개
- 숫자 최소 1개
- 허용된 특수문자 최소 1개

### 로그인 버튼 활성화 조건

다음 조건을 모두 만족할 때만 로그인 버튼을 활성화하고 `active` 클래스를 추가한다.

- 이메일 형식이 유효하다.
- 비밀번호 형식이 유효하다.
- 로그인 요청을 처리 중이지 않다.

### 로그인 성공 시나리오

1. 사용자가 유효한 이메일과 비밀번호를 입력한다.
2. 로그인 버튼이 활성화된다.
3. 사용자가 버튼을 누르거나 폼을 제출한다.
4. 브라우저의 기본 폼 제출을 막는다.
5. 이메일과 비밀번호를 다시 검증한다.
6. `isLoginSubmitting`을 `true`로 변경한다.
7. 로그인 버튼을 비활성화한다.
8. `POST /auth/login` 요청을 보낸다.
9. 요청에는 Access Token을 포함하지 않으며, 401 응답에도 토큰 재발급을 시도하지 않는다.
10. 응답의 `data.accessToken`을 메모리에 저장한다.
11. `pages/posts.html`로 이동한다.

### 중복 제출 방지

- 요청 시작 전에 `isLoginSubmitting`을 확인한다.
- 이미 요청 중이면 추가 제출을 무시한다.
- 요청 중에는 로그인 버튼도 비활성화한다.
- 요청 성공 또는 실패 후 `finally`에서 제출 상태와 버튼 상태를 복구한다.

### 회원가입 이동

1. 사용자가 `회원가입` 링크를 선택한다.
2. `pages/signup.html`로 이동한다.

### React 마이그레이션 시 보존할 동작

- 입력 중에는 버튼 활성 여부만 갱신하고, blur 시 구체적인 helper를 표시한다.
- 제출 시 이메일을 앞뒤 공백 없이 전송한다.
- 요청 중 중복 제출을 막는다.
- 로그인 요청에는 Access Token 및 자동 재발급을 적용하지 않는다.
- 서버 오류를 처리하기 전에 기존 helper를 모두 초기화한다.
- 성공 시 Access Token을 저장한 후 게시글 목록으로 이동한다.
- `label`, `autocomplete`, disabled 상태 등 기존 접근성 정보를 유지한다.

### 기준선에서 발견한 확인 필요 사항

- Access Token은 메모리에만 저장되므로 페이지를 이동하면 사라진다. 이후 보호 API가 Refresh Token 쿠키로 토큰을 재발급하는 것이 현재 의도된 흐름이다.
- 로그인 성공 응답에 `data.accessToken`이 없으면 `saveAccessToken()`에서 일반 JavaScript 오류가 발생한다. 이 경우 현재 화면에는 일반 로그인 오류 alert가 표시된다.
- 로그인 성공 후 이동 경로가 `posts.html` 상대 경로다. React에서는 `/posts`로 변경하되 사용자 관점의 결과는 동일하게 유지한다.
