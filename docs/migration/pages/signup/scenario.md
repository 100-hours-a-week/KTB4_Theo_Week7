# 회원가입 페이지 정상 작동 시나리오

## 1. 기준 소스

- 화면: `pages/signup.html`
- 동작: `js/signup.js`
- 공통 요청: `js/common.js`
- 검증 규칙: `js/validation.js`
- 스타일: `css/common.css`, `css/signup.css`

## 2. 페이지 정보

- 기존 URL: `pages/signup.html`
- React 예정 URL: `/signup`
- 진입 대상: 비로그인 사용자
- 성공 결과: 완료 alert를 표시하고 로그인 페이지로 이동한다.

## 3. 화면 구성

1. 서비스 로고와 태그라인
2. 로그인으로 돌아가는 뒤로가기 링크
3. 회원가입 제목과 설명
4. 필수 프로필 이미지 선택 영역
5. 이메일 입력
6. 비밀번호 입력과 형식 안내
7. 비밀번호 확인 입력
8. 닉네임 입력과 형식 안내
9. 회원가입 버튼
10. 로그인 페이지 전환 링크

## 4. 최초 진입 상태

- 선택된 프로필 파일은 없다.
- 기본 프로필 아이콘이 원형 선택 영역에 표시된다.
- 프로필 미리보기 이미지는 숨겨져 있다.
- 이메일, 비밀번호, 비밀번호 확인, 닉네임은 비어 있다.
- 모든 helper는 비어 있다.
- 회원가입 버튼은 비활성화되어 있다.
- 입력 필드의 자동완성 목적은 이메일 `email`, 두 비밀번호 필드 `new-password`다.

## 5. 프로필 이미지 시나리오

### 5.1 이미지가 없는 상태에서 선택 영역 클릭

1. 사용자가 프로필 이미지 영역을 선택한다.
2. 선택된 파일이 없으므로 `* 프로필 사진을 추가해주세요.`를 표시한다.
3. label의 기본 동작으로 파일 선택 창이 열린다.

### 5.2 이미지 선택

1. 사용자가 `image/*` 파일 하나를 선택한다.
2. 첫 번째 파일을 `selectedProfileFile`로 저장한다.
3. 이전 Object URL이 있다면 `URL.revokeObjectURL()`로 해제한다.
4. `URL.createObjectURL(file)`로 미리보기 URL을 생성한다.
5. 선택한 이미지를 원형 영역에 표시한다.
6. 기본 프로필 아이콘을 숨긴다.
7. 프로필 helper를 제거한다.
8. 폼 전체 유효성에 따라 버튼 상태를 갱신한다.

### 5.3 선택 이미지 제거

1. 이미지가 표시된 원형 영역을 다시 선택한다.
2. 파일 선택 창이 다시 열리지 않도록 label 기본 동작을 막는다.
3. 선택 파일과 file input 값을 초기화한다.
4. Object URL을 해제한다.
5. 미리보기를 숨기고 기본 프로필 아이콘을 다시 표시한다.
6. 프로필 필수 helper를 표시한다.
7. 회원가입 버튼을 비활성화한다.

### 5.4 현재 이미지 처리 제약

현재 API에는 실제 이미지 binary를 업로드하지 않는다. 선택된 `File`은 브라우저 미리보기에만 사용하며 요청에는 `file.name`만 `profileImage`로 전달한다.

## 6. 입력값 검증 시나리오

### 6.1 공통 검증 시점

- `input`: 구체적인 helper를 새로 표시하지 않고 전체 폼 유효성과 버튼 상태를 갱신한다.
- `blur`: 해당 필드 helper를 갱신하고 버튼 상태를 다시 계산한다.
- `submit`: 프로필 → 이메일 → 비밀번호 → 비밀번호 확인 → 닉네임 순서로 검증한다.
- 제출 검증은 `&&` 단락 평가를 사용하므로 앞 단계가 실패하면 뒤 단계 검증은 그 제출 시점에 실행되지 않는다.

### 6.2 이메일

- 앞뒤 공백을 제거한 값으로 검증하고 전송한다.
- 빈 값: `* 이메일을 입력해주세요.`
- 형식 오류: `* 올바른 이메일 주소 형식을 입력해주세요. (예: example@example.com)`

### 6.3 비밀번호

- 8자 이상 20자 이하
- 대문자, 소문자, 숫자, 허용 특수문자를 각각 최소 1개 포함
- 빈 값: `* 비밀번호를 입력해주세요.`
- 형식 오류: 비밀번호 규칙 helper 표시
- 비밀번호 blur 시 비밀번호 확인도 함께 다시 검증한다.

### 6.4 비밀번호 확인

- 빈 값: `* 비밀번호를 한번 더 입력해주세요.`
- 비밀번호와 불일치: `* 비밀번호가 다릅니다.`
- 일치하면 helper를 제거한다.

### 6.5 닉네임

- 앞뒤 공백을 제거한 값으로 검증하고 전송한다.
- 빈 값: `* 닉네임을 입력해주세요.`
- 문자열 내부에 일반 공백 포함: `* 띄어쓰기를 없애주세요.`
- 10자 초과: `* 닉네임은 최대 10자까지 작성 가능합니다.`

## 7. 버튼 활성화 조건

다음 조건을 모두 만족하고 요청 중이 아닐 때만 활성화한다.

- 프로필 파일이 선택되어 있다.
- 이메일 형식이 유효하다.
- 비밀번호 형식이 유효하다.
- 비밀번호 확인이 비밀번호와 같다.
- 닉네임이 비어 있지 않고 공백이 없으며 10자 이하다.
- 회원가입 요청 중이 아니다.

활성화 시 `active` 클래스를 추가한다.

## 8. 회원가입 성공 시나리오

1. 사용자가 모든 필수 값을 유효하게 입력한다.
2. 회원가입 버튼이 활성화된다.
3. 폼을 제출한다.
4. 중복 제출 상태를 확인한다.
5. 모든 필드를 순서대로 재검증한다.
6. `isSignupSubmitting`을 `true`로 바꾸고 버튼을 비활성화한다.
7. `POST /users/signup` 요청을 전송한다.
8. 로그인 요청과 마찬가지로 Access Token과 401 재발급을 사용하지 않는다.
9. 성공하면 `회원가입이 완료되었습니다.` alert를 표시한다.
10. 사용자가 alert를 닫으면 기존 `pages/login.html`, React에서는 `/login`으로 이동한다.

## 9. 중복 제출 방지

- 제출 시작 시 `isSignupSubmitting`이 이미 참이면 종료한다.
- 요청 중 회원가입 버튼을 비활성화한다.
- 성공 또는 실패 후 `finally`에서 제출 상태와 버튼 상태를 복구한다.

## 10. React 마이그레이션에서 보존할 동작

- 이미지가 필수인 현재 정책
- 선택 이미지 미리보기와 다시 클릭하여 제거하는 동작
- Object URL 생성과 해제
- input/blur/submit 검증 시점
- 기존 helper 문구와 표시 위치
- 유효한 폼에서만 버튼 활성화
- 요청 중 중복 제출 방지
- 로그인·회원가입 요청에 토큰 재발급을 적용하지 않는 정책
- 성공 alert 후 `/login` 이동
- 실패 후 입력값과 선택 이미지 유지
- label, accept, autocomplete 등 접근성·브라우저 속성
