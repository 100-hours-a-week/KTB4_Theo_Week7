# 게시글 상세 정상 작동 시나리오

## 1. 기준 소스

- 화면: `pages/post.html`
- 상세 동작: `js/post.js`
- Header 동작: `js/profile-menu.js`
- 공통 요청·표시 유틸리티: `js/common.js`
- 스타일: `css/common.css`, `css/post.css`

## 2. 페이지 정보

- 기존 URL: `pages/post.html?postId={id}`
- React 예정 URL: `/posts/:postId`
- 진입 대상: 로그인 사용자
- 목록 이동: `/posts`
- 수정 예정 URL: `/posts/:postId/edit`
- 잘못된 ID: alert 후 목록 이동

## 3. 화면 구성

1. 목록 뒤로가기, 서비스 로고, 프로필 메뉴가 있는 Header
2. 좋아요 통계 버튼
3. 제목, 작성자, 작성일, 수정 여부
4. 조회수와 댓글 수
5. 작성자 전용 수정·삭제 버튼
6. 선택적 첨부 이미지 갤러리
7. 줄바꿈을 보존하는 게시글 본문
8. 댓글 작성·수정 폼
9. 댓글 목록과 작성자 전용 수정·삭제 버튼
10. 게시글·댓글 공용 삭제 확인 모달

## 4. 진입과 상세 조회

1. Route param `postId`를 숫자로 변환한다.
2. 숫자가 아니거나 양의 정수가 아니면 `잘못된 게시글 주소입니다.` alert 후 `/posts`로 이동한다.
3. 보호 Route가 현재 사용자 인증을 초기화한다.
4. `GET /posts/{postId}`를 한 번 요청한다.
5. 요청 중에는 상세와 댓글 영역을 표시하지 않는다.
6. 성공하면 응답으로 상세, 갤러리와 댓글을 함께 렌더링한다.

React에서 ID가 아예 없는 `/posts`는 잘못된 상세 주소가 아니라 게시글 목록 Route다. 이는 query parameter를 사용하던 Vanilla와 Route 구조가 달라진 결과다.

## 5. 일반 게시글 표시

- 제목과 본문은 응답 문자열을 그대로 표시한다.
- 본문 줄바꿈은 `white-space: pre-wrap`으로 보존한다.
- 작성자가 탈퇴했으면 닉네임을 `알 수 없음`으로 표시한다.
- 작성자 이미지가 없거나 로드에 실패하면 기본 이미지를 표시한다.
- `createdAt`은 `T`를 공백으로 바꾸고 밀리초를 제외하여 표시한다.
- `edited === true`이면 `(수정됨)`을 표시한다.
- 좋아요, 조회수, 댓글 수는 1,000 이상일 때 `{n}k` 형식으로 표시한다.
- `liked`에 따라 좋아요 버튼의 `liked` class와 `aria-pressed`를 설정한다.
- `author === true`일 때만 게시글 수정·삭제 버튼을 표시한다.

## 6. 숨김 게시글

`blinded === true`이면 다음처럼 제한한다.

- 제목과 본문을 모두 `숨김 처리된 게시글입니다.`로 표시한다.
- 좋아요 버튼을 숨기고 비활성화한다.
- 이미지 갤러리를 표시하지 않는다.
- 댓글 폼, 댓글 목록과 댓글 섹션 전체를 표시하지 않는다.
- 수정·삭제 버튼은 현재 Vanilla에서 `author` 값만 기준으로 하므로 응답에 따라 표시될 수 있다.

## 7. 이미지 갤러리

### 이미지 없음

- `imageUrls`가 배열이 아니거나 유효한 값이 없으면 갤러리를 숨긴다.
- 이미지 `src`를 제거한다.

### 이미지 1개

- 첫 이미지를 표시한다.
- 이전·다음 버튼과 `1 / 1` indicator를 모두 숨긴다.

### 이미지 여러 개

- 최초 `1 / N` 상태로 시작한다.
- 첫 이미지에서는 이전 버튼을 비활성화한다.
- 마지막 이미지에서는 다음 버튼을 비활성화한다.
- 이전·다음 선택 시 범위를 벗어나지 않고 indicator를 갱신한다.
- 이미지 로드 실패 시 `image_fallback.png`로 교체한다.
- fallback도 실패하면 `src`를 제거하고 갤러리를 숨긴다.

## 8. 댓글 목록

- `comments`가 배열이 아니거나 숨김 게시글이면 빈 배열로 처리한다.
- 댓글이 없으면 별도 빈 상태 문구를 표시하지 않는다.
- 댓글마다 프로필, 닉네임, 작성일과 내용을 표시한다.
- 탈퇴한 작성자의 닉네임은 `알 수 없음`이다.
- `commentDeleted === true`이면 내용을 `삭제된 댓글입니다.`로 표시한다.
- 댓글 작성자이며 삭제·탈퇴 상태가 아닐 때만 수정·삭제 버튼을 표시한다.

## 9. 좋아요

1. 숨김 게시글이거나 요청 중이면 실행하지 않는다.
2. 버튼을 비활성화하고 `POST /posts/{postId}/likes`를 요청한다.
3. 성공 응답의 `liked`, `likeCount`로 버튼과 수치를 갱신한다.
4. 실패하면 기존 화면을 유지하고 오류를 표시한다.
5. 완료 후 버튼을 다시 활성화한다.

## 10. 댓글 작성

- textarea 최대 길이는 50자다.
- 공백 제거 후 빈 값이면 `* 댓글 내용을 입력해주세요.`를 표시한다.
- 50자 초과이면 `* 댓글은 최대 50자까지 작성 가능합니다.`를 표시한다.
- 입력값이 있을 때만 등록 버튼을 활성화한다.
- 요청 중 중복 등록을 차단하고 버튼을 비활성화한다.
- 성공하면 폼을 초기화하고 상세를 다시 조회한다.

## 11. 댓글 수정

1. 작성자 댓글의 수정 버튼을 선택한다.
2. 기존 내용을 상단 댓글 폼에 넣는다.
3. 버튼 문구를 `댓글 수정`으로 바꾸고 취소 버튼을 표시한다.
4. textarea에 포커스하고 폼이 화면 중앙에 오도록 스크롤한다.
5. 제출하면 `PATCH /posts/{postId}/comments/{commentId}`를 요청한다.
6. 성공하면 폼을 초기화하고 상세를 다시 조회한다.
7. 취소하면 빈 등록 폼으로 되돌린다.

## 12. 삭제 모달과 삭제

- 게시글 삭제: 제목 `게시글을 삭제하시겠습니까?`
- 댓글 삭제: 제목 `댓글을 삭제하시겠습니까?`
- 공통 설명: `삭제한 내용은 복구할 수 없습니다.`
- 열릴 때 확인 버튼에 포커스한다.
- 취소, backdrop 선택 또는 Escape로 닫는다.
- 삭제 요청 중에는 닫지 못하며 확인 버튼을 비활성화한다.
- 게시글 삭제 성공 후 `/posts`로 이동한다.
- 댓글 삭제 성공 후 모달을 닫고 상세를 다시 조회한다.

현재 Vanilla 모달은 focus trap과 닫은 뒤 trigger 포커스 복원을 구현하지 않는다. React 공통 `ConfirmModal`에서 이를 보강하면 접근성 개선으로 변경 기록에 남긴다.

## 13. React 마이그레이션에서 보존할 동작

- Route ID 검증과 잘못된 주소 처리
- 로딩 완료 전 상세·댓글 미표시
- 숨김·탈퇴·삭제 댓글 표시 규칙
- 작성자에게만 변경 버튼 표시
- 이미지 개수별 navigation과 fallback
- 수치·날짜 표시 형식
- 좋아요와 댓글 요청 중 중복 처리 방지
- 댓글 수정 시 상단 폼 재사용
- mutation 성공 후 상세 재조회 정책
- 삭제 모달의 대상별 문구와 닫기 조건
- API status/message별 alert와 이동
